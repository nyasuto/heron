## プロジェクト全体の最終ゴール

受動歩行機械の設計パラメータ空間を MAP-Elites で探索し、「どんな形・どんな歩き方の組み合わせが可能か」の地図を可視化する。

20年前の研究では「最強の1個」しか得られなかった景色を、Quality Diversity で「設計空間の全体像」として描き直す。

## Phase 1: Compass Gait Walker

**目的:** 「物理が合っている」ことを確認する。古典問題なので失敗パターンが既知。

膝なしの2脚倒立振子モデル。最も単純な受動歩行機。McGeer (1990) で解析的に扱われている古典。

### 完了条件

- [ ] 膝なし2脚倒立振子モデルが、3度傾斜の坂で歩く
- [ ] 少なくとも10歩、転倒せずに継続する初期条件が見つかる
- [ ] パラメータ（脚長、質量、足の半径、坂角度）を変える関数 or CLIがある
- [ ] 歩行軌跡（重心位置、関節角度の時系列）をログに出力できる
- [ ] 解析解と比較できる量を1つ持つ（推奨: 歩幅 or 歩行周期）

### 注意点

- **受動歩行は初期条件に極めて敏感。** 歩かない時の99%は物理バグじゃなく初期条件の問題。
- 安定極限サイクルに乗るには、適切な初期角度・初期角速度のセットが必要。
- 物理ステップは小さめ（1ms程度）にする。受動歩行は数値的に厳しい。

### このフェーズで決めること

- ロボット記述方法（URDF / Genesis ネイティブ API / 自前dataclass）
- 状態ログのフォーマット（CSV / JSON Lines / parquet）

## Phase 2: Kneed Walker

**目的:** ぽんぽこ殿が当時扱った研究対象本体に到達する。Phase 3でMAP-Elitesに渡す評価関数の入出力をここで固める。

膝関節を持つ受動歩行機。前進中は膝がロックされ、後退時にフリーになる。よりリアルな人間の歩行に近い。

### 完了条件

- [ ] 膝関節（前進中ロック、後退時フリー）を持つモデルが歩く
- [ ] Compass Gait と同じ実行インターフェースで動く（`simulate(params, slope)` 形式）
- [ ] パラメータ空間（もも長、すね長、足長、各部の質量）が dataclass で明示されている
- [ ] パラメータをランダムにサンプルして100回シミュレートし、生存率と平均歩行距離をログに出せる
- [ ] 1回のシミュレーションが headless で十分高速に終わる（目標: 1秒以下、Phase 3で1万回回すため）

### 注意点

- 膝のロック機構の実装方法は Genesis API で複数の選び方がある（コンタクト + フリー関節 / プログラム的にロック切替 など）。**Claude Code は実装前に方式の選択肢を提示すること。**
- Phase 3で MAP-Elites に渡すため、評価関数は **副作用なし・状態を持たない** 純関数として書く。

### このフェーズで決めること

- WalkerParams の最終形（Phase 3でこれが Genotype になる）
- WalkResult に含める情報（Phase 3 で Behavior Descriptor 計算に使う）
